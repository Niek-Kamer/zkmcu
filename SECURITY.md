# Security

## Supported versions

This is pre-1.0 software. Only the latest `0.x` release gets security fixes.

| version | supported |
|---------|-----------|
| 0.1.x   | yes       |
| < 0.1   | no        |

## Reporting a vulnerability

Open a [GitHub security advisory](https://github.com/Niek-Kamer/zkmcu/security/advisories/new) with reproduction steps and the affected crate version. Default disclosure window is 90 days from first report, earlier if a patch ships sooner. For ambiguous issues (is-it-a-bug?), a public issue is fine.

## Threat model (v0.1.0)

zkmcu is a verify-only Groth16/BN254 SNARK verifier for microcontrollers. The intended deployment is a device that:

- holds a **trusted verifying key** baked into firmware at provisioning time (or loaded from a trusted channel),
- receives **untrusted proof + public-input bytes** over whatever transport the device uses (USB, BLE, NFC, cellular, whatever),
- decides something on-chain or on-device based on `verify() → bool`.

Under that model the verifier must:

1. Never produce `Ok(true)` for a proof the arkworks reference verifier would reject.
2. Never crash, hang, or exhaust memory on adversarial input.
3. Never accept a canonically malformed `Fr` (protects nullifier uniqueness for downstream users).

What it does **not** defend against:

- **Secret data in the verify path.** `substrate-bn` is not constant-time, verify duration varies observably with public-input Hamming weight. If your application flows a secret into any verify-path argument, zkmcu leaks it via timing. Don't do that.
- **Power / EM side-channels.** Unmeasured. Requires a ChipWhisperer-class lab setup. Tracked as future work.
- **G2 subgroup membership.** `substrate-bn` is trusted for G2 subgroup checks and this is untested by zkmcu directly. Historically a bug class in BN254 precompile implementations, on the audit list.
- **Untrusted VK.** An adversary who controls the VK can engineer the pairing check to accept a forged proof. The firmware must provision the VK from a trusted source.

## What is tested

- Adversarial suite in `crates/zkmcu-verifier/tests/adversarial.rs` (23 tests): parser truncation, off-curve / off-twist points, field-element overflow, `Fr` non-canonical encoding, mismatched public-input counts, exhaustive single-bit tampering of VK / proof / public bytes. No bit-flip mutation produces `Ok(true)` on any known-good vector.
- Property-based suite in `crates/zkmcu-verifier/tests/properties.rs` (6 properties × 256 cases).
- Cross-library consistency: every committed test vector is generated by `ark-groth16` and natively verified there before being written, the embedded path re-verifies the same bytes with `substrate-bn`.

## Fixed in 0.1.0

### DoS via unbounded allocation

`parse_vk` and `parse_public` previously called `Vec::with_capacity(n)` where `n` came from untrusted input. An attacker sending `num_ic = u32::MAX` triggered a `u32::MAX × 96 B ≈ 412 GB` allocation, SIGABRT on desktop and instant reset on MCU. Patched with checked arithmetic plus buffer-length validation before allocation.

### `Fr` non-canonical encoding accepted

`substrate-bn::Fr::from_slice` silently reduces 256-bit inputs mod `r` instead of rejecting non-canonical encodings. Pairing correctness is unaffected (reduction preserves the pairing result), but this introduces malleability for any application that uses the raw `Fr` bytes as an identity (nullifiers, replay-protection tags, Merkle leaves). Patched with a strict `< r` check in `read_fr_at` before delegating to `substrate-bn`.
