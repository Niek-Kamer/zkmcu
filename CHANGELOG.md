## [0.1.0] — 2026-04-22

First public release. The three library crates (`zkmcu-verifier`, `zkmcu-vectors`, `zkmcu-host-gen`) share this version. The two firmware crates (`bench-rp2350-m33`, `bench-rp2350-rv32`) are not published to crates.io, they live in the repo for reproducibility.

Running on a Raspberry Pi Pico 2 W at 150 MHz, directly on-device:

- Groth16 / BN254 verify (1 public input, `substrate-bn` 0.6, unmodified): **~962 ms on Cortex-M33, ~1341 ms on Hazard3 RV32**.
- Peak stack: ~15.6 KB on both cores.
- Peak heap: ~79.4 KB (same for 1-input and 5-input circuits, the `pairing_batch` workspace dominates).
- Heap arena confirmed sufficient at **96 KB** with 17% margin, total RAM in use ≈ 111 KB, puts the verifier on the 128 KB SRAM MCU / secure-element class.
- Iteration-to-iteration variance under 0.07%.

Full per-run results in `benchmarks/runs/` as TOML + raw serial logs.

### Added

- `zkmcu-verifier`: `no_std` Rust Groth16 / BN254 verifier over `substrate-bn`.
  - `verify_bytes(vk, proof, public)` one-shot API.
  - `parse_vk`, `parse_proof`, `parse_public`, `verify` piecemeal API for when you want to verify many proofs against a single VK.
  - EIP-197 compatible wire format (32-byte big-endian field elements, `G1 = x ‖ y`, `G2 = x.c1 ‖ x.c0 ‖ y.c1 ‖ y.c0`, identity points encoded as all-zero).
- `zkmcu-vectors`: `no_std` test-vector loader exposing the `square` (1 public input) and `squares-5` (5 public inputs) Groth16 vectors as baked-in binary data via `include_bytes!`.
- `zkmcu-host-gen`: host-side CLI that uses `arkworks` 0.5 to produce the EIP-197-format `.bin` files consumed by `zkmcu-vectors`.
- `bench-rp2350-m33` / `bench-rp2350-rv32`: firmware that benchmarks the verifier on both cores of the RP2350. Includes stack painting, peak-heap tracking, and per-iteration verify timing over USB-CDC serial.

### Security

- Strict canonical encoding check on `Fr` (rejects values ≥ scalar modulus), closes a malleability hole left by upstream `substrate-bn`.
- DoS-hardened parsers: `num_ic` / `count` fields are validated against the buffer length before any allocation, prevents ~412 GB-allocation attacks from a malicious VK.
- Adversarial test suite (`zkmcu-verifier/tests/adversarial.rs`, 23 tests) plus property-based tests (`tests/properties.rs`, 6 properties × 256 cases) cover parser truncation, off-curve / off-twist points, field-element overflow, `Fr` non-canonical encoding, mismatched public-input counts, and exhaustive single-bit tampering of VK / proof / public bytes. No bit-flip mutation produces `Ok(true)` on any known-good vector.
- Cross-library sanity: every committed test vector is generated by `ark-groth16` and natively verified with arkworks before being written, the embedded path re-verifies the same bytes with `substrate-bn`.

Out of scope for 0.1.0, explicitly:

- Constant-time execution. `substrate-bn` is not CT, verify duration varies observably with public-input Hamming weight. Acceptable for verify-only threat models where proof and public inputs are already public. Not acceptable if secret data ever flows into the verify path.
- Power / EM analysis. Unmeasured, requires a lab setup. Separate follow-up.
- G2 subgroup membership. Assumed from `substrate-bn`, untested by zkmcu directly. Historically a bug class in BN254 precompile implementations, on the audit list.

See `SECURITY.md` for the full threat model.

### Notes

- On the RP2350 Hazard3 core, `mcountinhibit[CY]` is set at reset, so `mcycle` reads zero until a single `csrw mcountinhibit, zero`. Undocumented in cryptographic-benchmarking contexts as far as I can tell.
- Hazard3 beats Cortex-M33 by ~35% on G1 scalar multiplication (register-count advantage) but loses by 33% on pairing and 39% on overall verify. First public cross-ISA measurement of pairing-grade arithmetic on identical silicon that I know of.
