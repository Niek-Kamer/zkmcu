# 2026-04-21 — M33 Groth16 baseline

First working end-to-end Groth16/BN254 verify on the RP2350 Cortex-M33 core using unmodified `substrate-bn` 0.6.0 — no hand-tuning, no DSP intrinsics, no platform-specific asm.

## What was run

- Firmware: `crates/bench-rp2350-m33`
- Test vector: `crates/zkmcu-vectors/data/square` (`x^2 = y`, generated by `zkmcu-host-gen` via `ark-groth16` 0.5.0)
- Host cross-check (native): arkworks-generated proof parsed with `zkmcu-verifier`, verified with `substrate-bn::pairing_batch` — `ok=true`, tampered input rejected.

## Headline

**Groth16 verify: ~988 ms per call, stable to <0.1 ms.**

## Observations

- **G1/G2 variance is Hamming-weight dependent.** Scalars with fewer set bits finish ~3× faster on the same code path. Not a concern for verify itself (vk_x's scalar is the public input, so bound by the application, not the attacker).
- **Pairing on a non-identity G2 point is ~8% slower than on `G2::one()`.** Earlier standalone benchmark (with `q = G2::one()`) hit 495 ms; this run uses `q = G2::one() * s` and sees 533 ms. Likely substrate-bn fast-paths the canonical generator.
- **Multi-pairing savings are significant.** Four separate pairings would be ~2.1 s; the `pairing_batch` (Miller × 4 + shared final exp) takes 988 ms including the G1 linear combination for `vk_x`. ~2× a single pairing, not ~4×.
- **No heap fragmentation issues.** 256 KB heap, parse-once-at-startup pattern, no reallocations inside the hot loop.

## Anomalies

- First iteration's output line was corrupted to `[1] pag1mul start` — USB bulk-IN NAK during the first long-running pairing while the Linux host hadn't fully attached. Harmless; clean from iteration 2 onward.

## Reproduction

```bash
cargo run -p zkmcu-host-gen --release          # (re)generate the test vector
cd crates/bench-rp2350-m33
cargo build --release
scp target/thumbv8m.main-none-eabihf/release/bench-rp2350-m33 pi:/tmp/bench-m33.elf
# On Pi 5 with Pico in BOOTSEL:
picotool load -v -x -t elf /tmp/bench-m33.elf
cat /dev/ttyACM0 | tee raw.log
```
