# 2026-04-22 — Cortex-M33 BLS12-381 Groth16 baseline

First on-device Groth16/BLS12-381 verify on the RP2350 Cortex-M33 core.
Same silicon, same clock, same firmware harness as the BN254 baseline
(2026-04-21-m33-groth16-baseline), substrate-bn swapped for zkcrypto
bls12_381 0.8. Vectors at EIP-2537 wire format, generated by the
extended host-gen CLI and cross-checked against the same arkworks
reference before the bytes hit disk.

## Variance

7 iterations per op. Max-min over iterations on the full Groth16 verify
is *192,974 cycles* on a 302M median — variance of **0.064 %**, slightly
better than the BN254 baseline's 0.07 %. Every iteration returned
`ok=true`. No flakiness.

## Prediction comparison

Full numeric comparison in
`research/reports/2026-04-22-bls12-381-results.typ`. Predictions
committed in `research/reports/2026-04-22-bls12-381-prediction.typ`
are not edited here (the prediction doc is immutable by project
policy).

Four predictions missed outside the stated error bound:

- *G1 scalar mul*: predicted ~275 ms, measured *847 ms* (+208 %).
- *Pairing*: predicted ~1220 ms, measured *607 ms* (−50 %, faster
  than predicted).
- *Peak heap*: predicted ~130 KB, measured *77.5 KB* (−40 %).
- *Full verify*: predicted 2100–2400 ms, measured *2015 ms*, just
  below the stated low end.

G2 scalar mul matched prediction to within noise (predicted ~525 ms,
measured 523 ms).

## Tier alignment

Peak heap 79.4 KB + 96 KB arena with 17 % margin is sufficient (though
this firmware runs with a 256 KB arena since it's a first-run
de-risking build). Stack 19 KB + heap 96 KB = ~115 KB, directly
comparable to BN254's 111 KB. **The 128 KB SRAM tier story holds for
BLS12-381** — same chip class as the BN254 path
(nRF52832, STM32F405, Ledger ST33, Infineon SLE78).

Note the heap/stack split differs between the two curves:

| | BN254 (substrate-bn) | BLS12-381 (zkcrypto) | Δ |
|-|---:|---:|---:|
| Peak stack | 15.6 KB | 19.4 KB | +3.8 KB (stack) |
| Peak heap  | 81.3 KB | 79.4 KB | -1.9 KB (heap)  |
| Total      | 96.9 KB | 98.8 KB | +1.9 KB         |

zkcrypto's pairing uses stack-allocated `G2Prepared` line coefficients
where substrate-bn heap-allocates Fq12 polynomial workspace.

## Constant-time observation

BLS12 G1 scalar mul on zkcrypto varies by *0.09 % across 7 iterations
with random scalars*. By contrast, BN254 G1 scalar mul on substrate-bn
spans ~2× (30-59 ms depending on scalar Hamming weight).

This suggests zkcrypto's `G1Affine::generator() * Scalar` uses a
scalar-oblivious implementation (probably Joye's double-and-always-
add-ladder or a window-NAF where every window is processed
uniformly). **Not formally constant-time** — needs dudect / ChipWhisperer
follow-up — but a useful informal security-positive.

## Follow-ups worth measuring

1. *Heap-tuned rebuild at 96 KB arena.* Methodology parity with the
   BN254 path; predict: runs clean; commit as
   `2026-XX-XX-m33-bls12-heap-tuned/`.
2. *.text / .data / .bss sizes.* Run `llvm-objcopy --strip-debug` +
   `size` on a stripped ELF to get the flash footprint figure. Expect
   ~150 KB `.text` based on scaling.
3. *Disassembly diff vs BN254* for one Fp multiplication, to
   attribute the cross-ISA flip on G1 mul (measured on RV32, see
   `2026-04-22-rv32-bls12-baseline/notes.md`).

## Reproduction

```bash
# Dev host:
cargo build -p bench-rp2350-m33-bls12 --release
scp target/thumbv8m.main-none-eabihf/release/bench-rp2350-m33-bls12 \
    pid-admin@10.42.0.30:/tmp/bench-m33-bls12.elf

# Pi 5, Pico in BOOTSEL:
picotool load -v -x -t elf /tmp/bench-m33-bls12.elf
cat /dev/ttyACM0
```
