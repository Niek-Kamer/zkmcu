## In five lines

Bring in one of the two verifier crates, feed three byte buffers, get a `bool`.

```rust
use zkmcu_verifier::verify_bytes;                   // BN254 / EIP-197
// or
use zkmcu_verifier_bls12::verify_bytes;             // BLS12-381 / EIP-2537

let ok = verify_bytes(&vk_bytes, &proof_bytes, &public_bytes)?;
assert!(ok, "proof failed to verify");
```

Both crates expose the same five-function API (`parse_vk`, `parse_proof`, `parse_public`, `verify`, `verify_bytes`). Different curve, different wire format, same shape. See [Getting started](/getting-started/) for the reusable parser variant and the embedded-target integration.

<Aside type="note" title="Why this matters">
  Every current SNARK verifier ships for phones or servers. Hardware wallets, offline credential validators, IoT attestation tags — nothing runs SNARK verification on-device today because no open-source no_std verifier existed. zkmcu closes that gap on any 128 KB SRAM-class MCU, for both Ethereum's BN254 precompile format and BLS12-381's EIP-2537 format.
</Aside>

## What it's for

<CardGrid>
  <Card title="Hardware wallets" icon="laptop">
    Local ZK verification on the secure element, so a compromised phone can't trick the hw wallet into rubber-stamping a bogus Semaphore / Tornado / Aztec-style action.
  </Card>
  <Card title="Offline credentials" icon="seti:lock">
    Transit turnstiles, festivals, borders. Verify a privacy-preserving credential fully offline, no server round-trip.
  </Card>
  <Card title="IoT attestation" icon="linux">
    Small devices verifying each other's SNARK-attested provenance directly, without a cloud relay in the loop.
  </Card>
  <Card title="Cross-chain proofs" icon="external">
    BLS12-381 sync-committee proofs, Filecoin PoSt / PoRep, Aleo proofs — anything on a BN254 or BLS12-381 Groth16 stack.
  </Card>
</CardGrid>
