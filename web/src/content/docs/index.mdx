## In five lines

Bring in one of the verifier crates, feed the proof bytes, get a result.

```rust
use zkmcu_verifier::verify_bytes;                   // BN254 / EIP-197
// or
use zkmcu_verifier_bls12::verify_bytes;             // BLS12-381 / EIP-2537
// or
use zkmcu_verifier_stark::fibonacci::verify;        // winterfell STARK, Fibonacci AIR

let ok = verify_bytes(&vk_bytes, &proof_bytes, &public_bytes)?;
```

All three verifiers: `no_std`, pure Rust, one function call, measured on silicon.

<Aside type="note" title="Why this matters">
  Every current SNARK / STARK verifier ships for phones or servers. Hardware wallets, offline credential validators, IoT attestation tags, nothing runs ZK verification on-device today because no open-source `no_std` verifier existed. zkmcu closes that gap on any 128 KB SRAM-class MCU, for *all three* dominant proof-system families: Ethereum's BN254 precompile format, BLS12-381's EIP-2537 format, and winterfell's Goldilocks-based STARK.
</Aside>

## What it's for

<CardGrid>
  <Card title="Hardware wallets" icon="laptop">
    Local ZK verification on the secure element, so a compromised phone can't trick the hw wallet into rubber-stamping a bogus Semaphore / Tornado / Aztec-style action. Timing-deterministic under TLSF, side-channel resistance baked in.
  </Card>
  <Card title="Offline credentials" icon="seti:lock">
    Transit turnstiles, festivals, borders. Verify a privacy-preserving credential fully offline, no server round-trip. Both Groth16 and STARK verify paths covered.
  </Card>
  <Card title="IoT attestation" icon="linux">
    Small devices verifying each other's SNARK-attested provenance directly, without a cloud relay in the loop. STARK verify's 75 ms speed makes per-packet verification viable.
  </Card>
  <Card title="Cross-chain / zkVM proofs" icon="external">
    BLS12-381 sync-committee proofs, Filecoin PoSt, Aleo proofs, winterfell / Miden VM traces, anything on a BN254, BLS12-381, or Goldilocks-STARK stack.
  </Card>
</CardGrid>

## What makes it a family

<CardGrid>
  <Card title="Same repo shape" icon="open-book">
    Three verifier crates, one per proof system. Parallel API shape, same test-vector machinery, same firmware-bench template. Write once against `zkmcu_verifier::verify_bytes`, swap the curve later if you need to.
  </Card>
  <Card title="Same 128 KB silicon tier" icon="document">
    BN254 Groth16: 97 KB total RAM. BLS12-381 Groth16: 116 KB. STARK: 100 KB. All measured on the same RP2350 Cortex-M33 at 150 MHz. Any hardware-wallet-grade chip (nRF52832, STM32F405, Ledger ST33, Infineon SLE78) can run any of the three.
  </Card>
  <Card title="Deterministic timing" icon="approve-check-circle">
    With the production TlsfHeap allocator, STARK verify iteration-to-iteration variance sits at **0.08 %**, the silicon noise floor. That's side-channel-resistant verify without a custom-built constant-time implementation. [How](/determinism/)
  </Card>
  <Card title="Two ISAs, one source" icon="rocket">
    ARM Cortex-M33 and RISC-V Hazard3 on the same RP2350 chip. Byte-identical crypto source across both, only linker scripts and cycle-counter reads differ. The cross-ISA "pure crypto" performance ratio, measured: 1.21× for STARK, 1.33× for BN254 Groth16, 2.56× for BLS12-381.
  </Card>
</CardGrid>
