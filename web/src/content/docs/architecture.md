## Three library crates

| Crate | Role |
|---|---|
| `zkmcu-verifier` | The `no_std` verification algorithm. Owns parsing and the pairing check. Depends on `substrate-bn`. |
| `zkmcu-vectors` | `no_std` loader for committed binary test vectors via `include_bytes!`. |
| `zkmcu-host-gen` | Host-only CLI. Uses `arkworks` to generate Groth16 proofs and serialise to EIP-197 binary format. |

## Two firmware crates

| Crate | Target |
|---|---|
| `bench-rp2350-m33` | ARM Cortex-M33, `thumbv8m.main-none-eabihf` |
| `bench-rp2350-rv32` | RISC-V Hazard3, `riscv32imac-unknown-none-elf` |

Both firmwares import the same three library crates. The crypto source is byte-for-byte identical between them; only entry macros, linker scripts, and cycle-counter reads differ. The portability claim is what this layout makes testable.

## Verification algorithm

Given a verifying key `(α, β, γ, δ, IC)`, a proof `(A, B, C)`, and public inputs `x[0..n]`:

1. Compute `vk_x = IC[0] + Σ x[i] · IC[i+1]` in `G1`
2. Check that the product `e(-A, B) · e(α, β) · e(vk_x, γ) · e(C, δ)` equals the identity element in the target group `Gt`

All four pairings are evaluated in a single `substrate_bn::pairing_batch` call — the four Miller loops share the expensive final exponentiation. Total cost is roughly 2× a single pairing rather than 4×.

## Cross-library consistency

Every committed test vector under `crates/zkmcu-vectors/data/` is:

1. Generated by `ark-groth16` 0.5 on the host
2. Natively verified by `arkworks` before being written to disk
3. Re-verified on the embedded side by `zkmcu-verifier` (which uses `substrate-bn`)

The integration tests in [`zkmcu-verifier/tests/verify_square.rs`](https://github.com/Niek-Kamer/zkmcu/blob/main/crates/zkmcu-verifier/tests/verify_square.rs) load these bytes and call `verify`. Passing means two independent cryptographic libraries agree on both wire format and arithmetic. It's the main safety net against encoding drift between arkworks and substrate-bn.

## Why `substrate-bn`

- Already `no_std`-compatible out of the box
- Pure Rust, no C or assembly dependencies to cross-compile
- Ethereum-compatible, used in production Ethereum client implementations
- Smaller binary footprint than `ark-bn254` for a single-curve verifier

Alternative considered: `ark-bn254`. Works fine, but pulls in the whole arkworks generic-programming machinery — noticeably heavier compile times and bigger output binaries for a narrow verify-only use case. Substrate-bn is the leaner choice here.

## Workspace layout at the repo root

```
bindings/
├── crates/
│   ├── zkmcu-verifier/      ← no_std Groth16 verify
│   ├── zkmcu-vectors/       ← no_std test-vector loader
│   ├── zkmcu-host-gen/      ← host-only arkworks CLI
│   ├── bench-rp2350-m33/    ← Cortex-M33 firmware
│   └── bench-rp2350-rv32/   ← Hazard3 firmware
├── benchmarks/runs/<date>-<slug>/   ← raw.log + result.toml per run
├── research/                 ← Typst sources → PDFs (whitepaper, survey, reports)
└── web/                      ← this site
```
