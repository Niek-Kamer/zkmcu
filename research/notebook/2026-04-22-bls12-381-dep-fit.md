# 2026-04-22 вЂ” bls12_381 dep-fit spike

First research task towards extending zkmcu from BN254 to BLS12-381 Groth16
verification. This spike answers one question only: **does
`bls12_381` (zkcrypto, crates.io) build no_std for the two embedded
targets we care about**, or will we need to vendor / patch the crate
before the project can proceed.

Not measured in this spike: pairing time, verify time, any optimization
decisions. Those are phase 2.

## Hypothesis (written before running anything)

- **Primary**: `bls12_381 = "0.8"` with `default-features = false, features
  = ["groups", "pairings", "alloc"]` will build cleanly (`cargo build
  --release`) for both `thumbv8m.main-none-eabihf` and
  `riscv32imac-unknown-none-elf`. **Confidence: high.**
- **Rationale**: the zkcrypto crates have no `std` feature at all (so
  there's no implicit std dependency to disable), the `default` feature
  set is already no_std-friendly, and every transitive non-dev dep
  (`ff`, `subtle`, `rand_core`, optional `group`/`pairing`/`zeroize`) is
  from the same ecosystem and maintained to the same no_std standard.
- **Expected failure modes**:
  - `getrandom` pulled in transitively via a test/dev path could fail
    with "target not supported" on embedded. Low probability because
    `rand_core` is the non-dev dep and `getrandom` shouldn't be on the
    normal graph.
  - A transitive dep enabling an `std` / `alloc` feature we didn't ask
    for.
  - Some inline asm or `core_arch` guard for x86/aarch64 that the crate
    forgot to gate (never seen in zkcrypto but possible).

## Method

Minimal in-tree spike crate at
`research/notebook/2026-04-22-bls12-381-dep-fit/`, deliberately its own
workspace root (empty `[workspace]` table) so the parent workspace's
`[patch.crates-io]` table and member list don't interfere.

```
# Commands run (from the spike dir)
cargo build --release --target thumbv8m.main-none-eabihf
cargo build --release --target riscv32imac-unknown-none-elf
```

Success criterion: both commands exit 0 with no warnings about std /
target / missing symbols.

`cargo check` would be faster but doesn't codegen, so wouldn't catch
target-specific lowering issues (inline asm guarded wrong, intrinsics
missing). `cargo build` on a lib crate produces `rlib` вЂ” no link step,
so missing-symbol issues stay latent. That's acceptable for a dep-fit
spike; the link-time story is covered when this lands in a bench
firmware crate.

### rlib sizes (unlinked, with generic code pre-monomorphization)

| crate | M33 | RV32 |
|---|---:|---:|
| `bls12_381` | 1,271,928 B | 1,271,980 B |
| `subtle` | 131,452 B | 131,420 B |
| `rand_core` | 118,234 B | 118,214 B |
| `group` | 89,354 B | 89,294 B |
| `ff` | 62,670 B | 62,646 B |
| `pairing` | 13,892 B | 13,872 B |

The byte-for-byte parity between M33 and RV32 rlibs (differences only in
rustc hash bytes, ~0.0%) is itself a signal: the crate isn't doing any
arch-specific specialization, it's pure generic Rust leaning on the
compiler. That means we'll get whatever LLVM does for each target,
no hand-tuning from the crate itself. Matches the `substrate-bn`
baseline situation on BN254.

The 1.27 MB bls12_381 rlib is pre-link; after LTO into a bench firmware
with one verify entry point, `.text` will be much smaller. (BN254
verifier came in at ~73 KB `.text` under the same profile вЂ” expect
BLS12-381 to land in roughly the same order of magnitude, possibly 2Г—
because Fq12 tower over a larger prime.)

### Surprise worth recording

Initial spike lib.rs called `gt.is_identity()` without importing the
`group::Group` trait. Compile error, not a dep-fit issue. Worth noting
because: `bls12_381::Gt` inherits `is_identity` from `group::Group`
*only when you import the trait*. A full verifier built on this crate
will need `use group::Group;` and probably `use group::Curve;` at top
of the verify module. Minor ergonomic paper-cut, no architectural
impact.

### What this means for the project

1. **BLS12-381 Groth16 verifier on MCU is wiring work, not research.**
   No fork, no patch, no nightly. Just pick up `bls12_381` and go.
2. **Same dep-quality bar as the BN254 baseline.** `substrate-bn` gave
   us no_std out of the gate; `bls12_381` does the same. Both ecosystems
   made the right call years ago.
3. **The two backends will be comparable.** BN254-substrate-bn and
   BLS12-381-zkcrypto are both pure-Rust, generic, no arch-specific
   tuning. Cross-library comparison will be clean.

## Next-phase hypothesis (prediction before measurement вЂ” do not edit
## these numbers after the BLS12-381 bench runs)

For the next milestone (first BLS12-381 primitive benchmarks on
RP2350 @ 150 MHz, same harness shape as the BN254 firmware), predictions:

- **Fq mul**: BLS12-381 is a 381-bit prime, BN254 is 254-bit. Schoolbook
  256Г—256 multiplication on a 32-bit core is 8Г—8=64 word-muls; 384Г—384
  is 12Г—12=144 word-muls. Ratio 2.25Г—, plus extra Montgomery-reduction
  cost. Prediction: **BLS12-381 Fq mul ~2.5Г— BN254 Fq mul**.
- **G1 scalar mul (typical)**: scalar is 255 bits for BLS12-381, 254
  for BN254, so NAF window count is similar. Each window op is
  dominated by Fq mul. Prediction: **~2.5Г— BN254 G1 mul в‰€ 275 ms on
  M33** (vs 110 ms BN254).
- **Pairing**: tower is Fq12 for both. Per-pairing Miller loop is
  ~65-70 doublings+additions on BLS12-381 (shorter than BN254's
  ate-loop because the BLS embedding degree choice), but each op is
  ~2.5Г— more expensive. Final exponentiation is heavier (easy part Г—
  hard part with cyclotomic). Prediction: **~2.3Г— BN254 pairing в‰€
  1.22 s on M33** (vs 533 ms BN254).
- **Full Groth16 verify (1 public input)**: 3 G1 scalar muls + 4
  pairings (batched) + pippenger on IC. Pairings dominate.
  Prediction: **~2.2-2.5Г— BN254 verify в‰€ 2.1-2.4 s on M33**
  (vs 962 ms BN254).
- **Memory**: Fq12 workspace for batch pairing is larger by a factor
  proportional to field size. Prediction: peak heap ~2Г— BN254 в‰€ 160 KB.
  Worth flagging up front because this *might push us over the 128 KB
  SRAM tier* and land us on the 256 KB tier. **If that happens, it
  changes the grant pitch.** рџ¤ћ it stays under.

These numbers are my guesses based on field-size scaling. I expect at
least one of them to be wrong by more than the stated bound вЂ” that's
the point of writing them down first.
